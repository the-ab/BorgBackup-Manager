Operations manual

Using BorgBackup Manager

This manual covers all features in version 1.0.47, from installation and sign-in through devices, repositories, jobs and schedules to archive management, restore, user permissions and diagnostics.

1. Quick start and system layout

The manager runs as a Docker container and controls Borg centrally. For a backup, the manager starts Borg on the selected client over SSH. The client reads its local files and writes to the configured repository. Clients do not need separate backup scripts or cron jobs.

Under Infrastructure, the sidebar now contains only Devices and System. The System workspace combines Notifications, Users, Manager backup, Settings and System diagnostics in one tab row. Diagnostics are no longer shown on the dashboard.

  1. Sign in as an administrator and add the public controller key to the client's authorized_keys.
  2. Add the device, verify its Ed25519 fingerprint and detect its Borg version.
  3. Create a managed or external repository, or import an existing local repository.
  4. Save a backup job with device, repository, sources, exclusions, compression and retention.
  5. Configure repository access directly on the backup job.
  6. Optionally assign a central schedule.
  7. Test the connection and run a small test backup.
  8. Browse an archive and test restore with a dry run first.
Important paths: manager data /data, repositories /repositories, run logs /data/run-logs, archive cache /data/archive-cache, manager Borg cache /data/borg-cache/repository-<ID>, device cache $HOME/.cache/borgbackup-manager/repository-<ID>, and security data /data/security.

2. HTTPS, sign-in and security data

Sign-in

  • Personal accounts with role-based access.
  • Passwords stored as salted scrypt verifiers.
  • Server-side revocable sessions with an HttpOnly cookie.
  • A tab-bound reload session as a secured fallback.
  • Source-scoped login rate limits without a globally abusable account lock.

Secrets

  • Repository passphrases, Borg keyfiles, and private SSH/TLS keys are encrypted with the master key.
  • /data/security/security.db contains users, sessions and encrypted secrets.
  • /data/security/master.key is the external trust anchor and must be backed up.

The initial password can be displayed with python -m app.initial_admin. Use ./recovery.sh in the project directory for account recovery, unlock and password reset.

3. Dashboard and active tasks

The dashboard shows job counts, running, queued and failed runs, plus repository count and total size. The Backup Jobs panel includes status, device, repository, sources, schedule, latest run, size of the latest successful backup, and a direct start button. It can be sorted by name, status, device, repository, latest run or backup size.

The status control next to the theme button always represents the current running task. Clicking it opens that task's live log directly, without an intermediate menu. If no task is running but one is queued, it opens the next queued run. After completion, the same area briefly shows the action result.

Attention highlights critical or outdated Borg versions, missing repository access and failed runs. System diagnostics are loaded, refreshed and closed under System → System diagnostics.

5. Repositories

Repositories can be sorted by name, readiness, type, job count or size. The choice is retained for the signed-in user in this browser. The table shows the numeric manager ID of each repository record directly beside its status; it is the same ID used in manager-side and device-side BBM cache paths such as repository-<ID>.

Managed

Stored below /repositories. Archive listing, check, prune, compact, diff and browsing run locally in the manager. Clients write through the integrated repository SSH service.

External

Uses an existing Borg URL. SSH keys and host key are managed centrally; no backup job is required to display archives.

  • Browse Directory detects existing local Borg repositories. Import Existing Local Repository never initializes or overwrites them.
  • Test Connection queues the Borg check, returns a run ID immediately, and stores diagnostics and size information in the run log. Long external checks therefore no longer produce an HTTP 504 timeout.
  • Clear Cache directly removes only the manager-side cache for the selected repository, never archives or repository configuration.
  • Compact can be started directly on the repository without a backup job; active repository runs and archive mounts block the action.
  • If a previously initialized managed repository no longer has a Borg config, the list shows Repository missing. Reset is allowed only for a completely empty target directory, never deletes files, and makes Initialize available again. Jobs and repository operations remain blocked until reinitialization succeeds.
  • Archive lists are persisted in /data/archive-cache and invalidated after write operations.
  • Original, compressed and deduplicated sizes are shown in a compact vertical layout.
  • The storage guard checks the actual filesystem of every mounted repository path. A global default and per-repository overrides are supported.

6. Backup jobs

The job list shows source size and file count. Values are updated exactly from completed Borg backups. A manual refresh performs a repository-independent live scan on the source device, counts the configured sources before Borg exclusions, and does not create an archive. A job connects one device to one repository and defines sources, exclusions, archive name template, compression, Borg create options and retention. The list can be sorted by name, status, device or repository, with the choice stored per user and browser. New archives use a compact permanently reserved prefix; historic prefixes remain assigned.

The archive browser behaves like a file browser with breadcrumb navigation and columns for size, type, permissions, owner/group and modification time. More opens a compact action bar grouped into Checks, Repository Access, Storage Maintenance and Management. It contains connection test, Borg check, job info, repository access, prune, compact, enable/disable, edit and delete. Access setup affects only this job's device/repository pair. Disabled jobs keep all settings and schedule assignments but are not started; active or queued runs must finish first.

Write operations for the same repository are serialized; different repositories may run in parallel. Manager commands use /data/borg-cache/repository-<ID>, while device commands use $HOME/.cache/borgbackup-manager/repository-<ID>. When SSH uses root, $HOME is /root; a path below /root/.cache is therefore a local device cache, not the repository. Backup, prune, compact and archive changes invalidate the archive cache. Compact uses verbose Borg output and reports the estimated freed space when Borg provides it.

7. Central schedules

Schedules are managed independently of jobs. They can target selected devices, selected jobs or all jobs in a repository; later matching jobs are included automatically. A job without assignment remains manual.

Daily, weekdays, weekends, selected weekdays, monthly and custom cron expressions are supported, including multiple times. Schedules can be enabled individually. After a schedule chain the repository size can be updated once; optional prune and compact may follow. Each schedule can also define its own concurrency cap: 0 uses only the global limit, while 1 queues jobs across different repositories one after another.

8. Runs, queue and live log

The Runs page filters by running, queued, successful, warning, failed or cancelled, with search and configurable list length. A run can be opened, cancelled, retried or deleted after completion. At most one Borg action runs per repository, while the global setting and optional schedule cap can also limit concurrency across different repositories. FIFO planning fills free slots with eligible runs so an older run that is itself blocked does not unnecessarily stop independent repositories. Queued logs identify the repository, schedule or global limit and, when available, the blocking run ID. Only live manager tasks consume slots, so orphaned run state cannot block the queue indefinitely. For remote backups, cancellation first closes the supervised control channel. The wrapper on the device then stops the complete Borg process group with SIGINT and confirms its exit before SSH and the queue slot are released; escalation to SIGTERM and SIGKILL is used only when the process does not respond. Automatic break-lock is intentionally avoided for repositories shared by multiple clients. The remote wrapper no longer requires the GNU-specific env --default-signal extension, so clients using BusyBox or older coreutils releases are supported as well.

The live dialog shows summary, trigger type, schedule, Borg version, saved backup sizes and readable output. Warning causes are collected line by line while Borg is running and are stored permanently on the run. Changed files (C), file access errors (E), disappeared paths, permission, I/O and pattern warnings therefore remain visible even when large amounts of output follow or the visible log is truncated. When the full file list is disabled, the manager still records warning-relevant C and E statuses. If Borg returns only rc 1 without a concrete detail line, the dialog explicitly shows “Cause not emitted”. Technical Details contains the command, stdout and filtered stderr.

Complete logs are stored in /data/run-logs/run-ID.log. Retention, maximum file size, UI display limit and cleanup are managed in system settings.

9. Archive list, filters, browser and export

  1. Select a repository and click Show Archives. The first request reads Borg; later requests use the persistent cache.
  2. Reload from Repository captures changes made outside the manager.
  3. Archives are always sorted newest first.
  4. Device resolution first uses current or historic archive series, then the Borg hostname and archive name. Generic formats such as docker-2026-07-17_03-20 are supported.
  5. Checkpoint archives can be included separately.
  6. Each row shows duration, files, original, compressed and deduplicated size; Details adds start, end, hostname, user, comment and archive ID.
  7. The browser works without FUSE and provides file-browser breadcrumbs, size, type, POSIX permissions, owner/group and modification time. A selection can be passed to restore or exported as TAR.GZ for a matching managed repository job.
  8. Administrators can select one or multiple archives and delete them in one repository-wide action. A mixed selection is labelled “Multiple devices”; Compact optionally runs once after all deletions.
  9. Restore and rename require an unambiguous backup-job assignment.

11. Manager backup and restore

A manager backup contains the manager and security databases, master key, settings, controller and repository keys, Borg keyfiles and TLS files. Repository data, Borg cache, archive cache and complete run logs are deliberately excluded.

  • New backups: encrypted .bbm files only, using AES-256-GCM and a non-stored passphrase of at least twelve characters.
  • Upload backup: select an existing .bbm or historical .zip file. File name, size limit and structure are validated; existing files are never overwritten and the stored file uses mode 0600.
  • Existing ZIP files: remain readable for legacy backups but can no longer be created.
  • Web UI restore: validates paths, entry count, sizes and compression ratio, creates an encrypted safety backup with a separate passphrase, restores data and restarts the container.
  • Server migration: restore with restore-backup.sh before the first start.

12. Notification center

Administrators can send backup, repository, schedule and other run events through SMTP, a generic JSON webhook, Discord or Telegram. Failures and warnings are selected by default; successful events can be enabled optionally.

  • Secrets: SMTP password, webhook URL and Telegram token are encrypted in the security database and are never returned to the Web UI.
  • Testing: every enabled channel has its own test action.
  • Delivery log: records success or the concrete delivery error without changing the Borg run result.
  • Diagnostic excerpt: filtered and bounded, and can be disabled completely.
  • Webhook: generic JSON containing event, severity, title, message, run ID and timestamp, or a Discord-compatible embed.

Delivery starts only after repository and concurrency queue slots have been released. A slow or unavailable notification service therefore does not hold up subsequent backups.

13. Users, roles and personal preferences

Administrator

Manages infrastructure, jobs, schedules, system settings, manager backups and users, plus all operational functions.

User

Read-only viewer role for the dashboard, lists and summarized run status, plus personal language and appearance. Operational actions, full logs, archives and configuration remain administrator-only.

Every user can open Appearance & Language and save their language (German or English) and theme (automatic, light or dark) individually. These values do not change a system-wide setting. The quick theme button also changes only the current user account.

Administrators can create users, change role and enabled status, set temporary passwords and require a password change. The current account and the last active administrator are protected.

15. Diagnostics, mobile view and safe operation

  • Sign-in: source-scoped rate limits protect expensive password verification without allowing third parties to lock an account globally.
  • Sessions: cookies are Secure, HttpOnly and SameSite=Strict, with absolute and idle expiry limits.
  • Proxy trust: forwarded headers are accepted only from explicitly configured proxy networks.
  • Readiness: /api/ready and the minimal status endpoint /api/health/strict; detailed components are administrator-only at /api/system/health.
  • System diagnostics: Borg version, repository base mount, all sub-mounts, usage, storage guard, sshd, forced commands and repository access. Access checks run directly as Web user borg; the root-only sshd -t check is handed off at container startup.
  • Repository port: allow port 2222 through the firewall only for known clients.
  • Repairs: automatic borg check --repair and break-lock are deliberately not offered.
  • Destructive actions: verify selection and backup state before prune, compact, archive deletion and restore.
  • Mobile view: Menu opens navigation; tables become cards, actions wrap cleanly and dialogs remain inside the viewport.
